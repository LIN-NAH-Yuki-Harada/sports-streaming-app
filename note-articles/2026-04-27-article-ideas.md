## ~~候補 B（機能予告 + ブランド構築 ミックス）~~ — 却下

「**もうすぐ、配信した試合が自動でアーカイブされるようになります — その理由を聞いてください**」

- 着想元: 今日（4/27）Sprint A 実装が完了したこと + 4/25 テスト配信で「アーカイブ機能の評価が非常に高い」というフィードバックが出たこと
- ペルソナ: B（撮影担当保護者）+ A（祖父母）両刺さり
- 戦略マッピング: ブランド構築型 ＋ 期待値醸成
- 効果: 既存ユーザーの定着 / 新規ユーザーの背中押し（「今登録すれば、すぐアーカイブ機能が使える」期待）
- なぜ推奨か: PMF シグナルと開発進捗を 1 本に重ねられる希少なタイミング。次の機能を「予告」する形は note の連載感を作りやすい

### 骨子案（800〜1200 字目安）

1. **導入**: 4/25 のテスト配信で、観てくれた保護者の方からこんな声をいただきました（具体エピソード or 抽象化した声）
2. **声**: 「試合中はリアルタイムで観られたけど、後から見返したい」「祖父母にも見せたいけど時差で見逃した」（実フィードバックを使う・編集 OK）
3. **解決**: 配信が終わると、自動的にアーカイブとして残るようにします
4. **仕組みの軽い説明**: 配信者の YouTube に unlisted でアップロード（URL を知っている人だけ視聴可能）、配信者本人がいつでも削除可能、容量や端末を圧迫しない
5. **タイミング**: もうすぐ、近々（具体日付は審査次第なので濁す）
6. **CTA**: 「先に登録しておいて、ローンチ第 1 弾で使ってみてください」+ live-spotch.com リンク

### 注意点

- 「YouTube」を強調しすぎると Google 審査前に外部誤解を招くリスク。「配信者ご自身のチャンネルに自動保存」と書くと安全
- 審査通過時期が確定していないので「来月」「来週」など断定的な日付は書かない
- フラグ ON 前なので「先行公開してます！」と勘違いを誘発しないよう「もうすぐ」のトーン徹底

## 候補 A: SEO 流入型（既に持ち越しタスク）

「**子供の試合をライブ配信、何を選べばいい？保護者目線で 4 つの選択肢を比較してみた**」

- 着想元: 既に memory `project_next_session_agenda.md` の持ち越しに記載済（note 第 2 本目「保護者の試合配信比較ガイド」執筆検討）
- ペルソナ: B（撮影担当保護者）/ E（進路選択保護者にも刺さる）
- 戦略マッピング: SEO 流入型
- ターゲット検索ワード: 「子供 試合 ライブ配信」「少年団 配信 アプリ」「部活 配信 比較」
- なぜ推奨候補か: 永続 SEO 資産になる。ただし「自社推し」が強すぎると比較記事として信頼を失う

### 骨子案

1. 比較対象: YouTube Live / Instagram Live / スポーツブル / LIVE SPOtCH
2. 比較軸: 配信のしやすさ / スコア表示 / 視聴側の手軽さ / 録画保存 / 料金 / 子供の出演同意
3. 表で見やすく
4. 結論: 用途別おすすめ（誰でも見られるなら YouTube、関係者限定なら LIVE SPOtCH 等）
5. ライブスポッチが推せる場面を素直に提示（押し付けない）

## 候補 C: 共感拡散型

「**離れて暮らす祖父母に、孫の試合を届けたかった**」

- 着想元: テスト配信時の保護者フィードバック / SNS ペルソナ A の主軸エピソード
- ペルソナ: A（祖父母・遠方親戚）軸足、B にも刺さる
- 戦略マッピング: 共感拡散型
- なぜ推奨候補か: note 公式マガジン採用や RT 拡散で当たりやすい型。実話ベースだと強い

### 骨子案

1. 導入: ある保護者からの相談（or オーナー自身の体験を抽象化）
2. なぜ既存サービスでは届かなかったか（YouTube Live のリンク共有が祖父母には難しい・通知が届かない・後から見られない、等）
3. 試合当日の出来事を時系列でドラマチックに
4. 共有された画面の様子、祖父母からの「観られたよ」LINE のスクショ感（再現テキストでよい）
5. ライブスポッチを作っている理由はこういう瞬間です、で締め
6. 注意: 個人特定要素は伏せる、許諾が取れる範囲で書く

## 候補 D: 業界論型（少し変化球）

「**個人開発者ひとりと AI で、地域のスポーツ配信プラットフォームを作っている話**」

- 着想元: 4/27 の memory `feedback_autonomous_mode.md`（Claude にコード作業を任せる開発スタイル）/ 今日 PostgreSQL の罠を Claude が発見・根治したエピソード
- ペルソナ: D（OB/OG）+ 業界関係者・メディア
- 戦略マッピング: 業界論型 + ブランド差別化
- 効果: 「個人だから細やかに対応できる」差別化、メディア寄稿元になる
- なぜ推奨候補か: 競合（スポーツブル等）との差別化を「人格」で出せる。Claude/AI 時代の文脈にも乗れる

### 骨子案

1. なぜひとりで作っているのか（地域のスポーツ配信は大手にとって採算合わない、保護者の声を直接聞ける距離感）
2. AI 開発エージェント（Claude）と二人三脚の実態（コード調査・実装・修正を任せる、私は判断と方針決め）
3. 具体エピソード: 4/27 にデータベースの権限設定の落とし穴を Claude が発見、本番影響ゼロのうちに修正できた
4. ひとりだからこそできること: ユーザーの声を翌日には反映できる、機能の優先順位を保護者目線で決められる
5. これからのロードマップを少しだけ
6. 注意: 「AI が全部やってる」ではなく「私が決めて AI が実装する」スタンス維持。技術を全面に出しすぎず、保護者向けの言葉に翻訳

### 注意点

- 自社サービスのバグの話は書き方を慎重に。「Claude が事前に見つけてくれた」というポジティブ展開なら OK、「漏れていました」と書くとリスク
- セキュリティ修正の詳細は書かず、「ある朝、配信機能の準備中に Claude が小さな見落としを指摘してくれた」程度の抽象化

## 候補 E: 実用ガイド型

「**保護者が試合のライブ配信でつまずく 5 つの「あるある」と、その全部の解決策**」

- 着想元: テスト配信運用で出てきた現場の悩み + 焼き込みスコアボード / 音声処理 / ヘルスバッジ等の機能群
- ペルソナ: B（撮影担当保護者）直撃
- 戦略マッピング: 実用ガイド型 + SEO 補助
- 効果: 「保存しておきたい」記事として note フォロー喚起。Pinterest 等への二次拡散も狙える
- なぜ推奨候補か: 機能宣伝にならず実用性で読まれる。SEO 検索ワードもいくつか拾える

### 骨子案

1. あるある①: 試合中に画面の前に出てしまって映像が遮られる → スコアの常時オーバーレイ表示で解決
2. あるある②: 撮影者の声がコートの音より大きすぎる → 自動で音量バランス調整
3. あるある③: 試合の途中で配信が落ちる・通信が悪くなる → 配信前のヘルスチェック
4. あるある④: スマホが熱くなって長時間配信できない → 発熱対策の最適化
5. あるある⑤: 試合終了後、見返したくなる → 自動アーカイブ機能（候補 B との連動）
6. ライブスポッチでは全部解決を目指してます、で締め
